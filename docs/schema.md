# Snapshot Schema Notes

Public v0 uses:

- envelope schema `octra-vitals-envelope-v0`;
- payload schema `octra-vitals-snapshot-v0`;
- evidence schema `octra-vitals-evidence-v0`;
- summary-row schema `octra-vitals-summary-row-v0`.

The envelope contains:

- `payload_hash`
- `evidence_manifest_hash`
- `canonical_payload`
- `canonical_evidence_manifest`
- `source_refs`
- `submitter`

`/api/latest` also exposes `payload` and `source_refs` at the top level for app consumers. Those fields mirror `envelope.payload` and `envelope.source_refs`; they are not a second canonical object.

The `record_snapshot_v0` transaction hash is receipt metadata and is not part of the pre-submit envelope.

## AML Write Path

The current Vitals State Program write path is one atomic AML call generated by:

```bash
npm run program:update
```

The hosted updater uses one process for collection, call construction, submission, AML readback, and receipt promotion. The lower-level `snapshot`, `program:record-call`, and `program:submit-snapshot` commands remain debug primitives.

The generated call stores the latest full payload, evidence manifest, and source refs, then rolls a bounded summary window for trend rendering:

```text
record_snapshot_v0(snapshot_id, observed_at, epoch_id, snapshot_index, payload_schema_version, summary_schema_version, payload_hash, evidence_hash, source_refs_hash, summary_hash, canonical_payload, canonical_evidence_manifest, canonical_source_refs, summary_row)
```

`record_snapshot_v0()` verifies:

- caller is owner/operator;
- program is not paused;
- `snapshot_id == "vitals." + observed_at`;
- `snapshot_index == snapshot_count + 1`;
- epoch/time move forward under the monotonicity rule;
- payload, evidence, source refs, and summary row hashes match domain-separated canonical strings;
- latest payload/evidence/source refs/window sizes stay within explicit caps.

The Vitals State Program intentionally does not store historical full payload maps, generic records, staged snapshot records, duplicate id maps, or an unbounded id index. Historical full bodies can be archived outside AML by content hash. AML history is the fixed recent summary window.

For the programmed Site Circle mainnet target, the same `record_snapshot_v0` semantics should live on the Site Circle's attached AML program. The state shape, hash domains, monotonicity checks, latest-body storage, and bounded summary window should remain the same. The write target changes from a separate State Program address to the Site Circle program, and the program must be initialized explicitly with `initialize_v0` after `program_update`.

## Summary Rows

Summary rows are fixed-width commitments to the fields the UI needs for trendlines:

- `snapshot_index`
- integer-second UTC `observed_at_unix`
- Octra epoch
- external block height
- issued, burned, encrypted, locked, wrapped, and unclaimed raw OCT values
- route count
- payload hash prefix

The public v0 window keeps up to 48 rows. Each row is 208 bytes, and the window is rolled with `substr()` + `concat()` so AML does not need loops or unbounded maps.

`get_latest_bundle()` returns a compact inspection string with latest index, id, observed time, epoch, payload/evidence/source-ref/summary hashes, summary-window hash, paused flag, successor flag, successor program, and the latest summary row. The summary row is last because it is also pipe-delimited. Full latest payload, evidence manifest, and source refs stay on dedicated getters to avoid turning the bundle into another large return surface.

The hashes are:

```text
payload_hash          = sha256("octra-vitals:snapshot:v0\n" + canonical_payload)
evidence_hash         = sha256("octra-vitals:evidence:v0\n" + canonical_evidence_manifest)
source_refs_hash      = sha256("octra-vitals:source-refs:v0\n" + canonical_source_refs)
summary_hash          = sha256("octra-vitals:summary:v0\n" + summary_row)
summary_window_hash   = sha256("octra-vitals:summary-window:v0\n" + summary_window)
```

AML cannot parse canonical JSON, so it cannot prove the summary row was faithfully extracted from the payload. The gateway reads both the latest payload and latest summary row, recomputes the row from the payload, and fails the latest read if they differ. After that check, historical rows remain hash-bound commitments.

## Bridge Accounting Fields

All OCT and wOCT amounts use 6 raw decimals. The collector now calls wOCT `decimals()` at the same pinned Ethereum block as `totalSupply()` and fails the snapshot if the returned value does not match the configured expected decimals (`VITALS_EXPECTED_WOCT_DECIMALS`, default `6`). This keeps the bridge equation from silently comparing differently scaled units.

Direct observed fields:

- `bridge.total_locked_oct_raw`: direct Octra RPC `octra_contractStorage(<BridgeVault>, "total_locked")`. Label as "Bridge ledger locked OCT".
- `bridge.total_unlocked_oct_raw`: direct Octra RPC `octra_contractStorage(<BridgeVault>, "total_unlocked")`. Label as "storage total_unlocked" until the vault source semantics are published.
- `bridge.vault_balance_oct_raw`: direct Octra RPC `octra_balance(<BridgeVault>).balance_raw`. Label as "Octra vault account balance".
- `bridge.lock_nonce` and `bridge.unlock_count`: direct Octra RPC storage counters. They are not token amounts.
- `bridge.woct_supply_raw`: direct Ethereum RPC `eth_call` of wOCT `totalSupply()` at the sampled Ethereum block, hex-converted to raw decimal. Label as "Ethereum wOCT supply".

Vitals-derived fields and displayed formulas:

- `bridge.unclaimed_oct_raw`: `sum(recovery.json.by_recipient[][].amount_raw)`. This is derived from relayer recovery entries and should be labeled "Relayer unclaimed claims". It is assigned to recovery recipients, not protocol-owned.
- `unclassified_locked_oct_raw`: `max(bridge.total_locked_oct_raw - bridge.woct_supply_raw - bridge.unclaimed_oct_raw, 0)`. This is not an upstream field. It is locked bridge collateral that the current snapshot cannot classify as either minted wOCT or relayer recovery claims.
- `vault_surplus_oct_raw`: `bridge.vault_balance_oct_raw - bridge.total_locked_oct_raw`. This is the vault account balance above the bridge storage ledger.
- `vault_unmatched_vs_claims_oct_raw`: `bridge.vault_balance_oct_raw - bridge.woct_supply_raw - bridge.unclaimed_oct_raw`. This is useful for audit review, but the app leads with the ledger-based unclassified value because `total_locked` is the bridge accounting ledger.

Each snapshot includes `payload.health.conservation`, a signed publish-and-flag summary of the same accounting checks. The producer fails closed before writing if required evidence is missing or unit scale is unsafe, including a missing Octra RPC `burned` field or a wOCT `decimals()` mismatch. For invalid accounting identities, it records the direct values and flags conditions such as `issued > max`, `wOCT + unclaimed > locked`, `vault_balance < locked`, or `max - issued != burned`.

Bridge health uses these rules:

- red: `issued > max`, `encrypted > issued`, `max - issued != burned`, `wOCT + unclaimed > locked`, `vault_balance < locked`, or wOCT decimals mismatch;
- yellow: reserved for future non-critical signed review states;
- green: required inputs are present, units are verified, burned reconciles to cap remainder, claims do not exceed locked collateral, and vault balance covers the locked ledger.

The bridge snapshot is not cross-chain atomic. Octra epoch, relayer finalized/scanned epoch, and Ethereum block are sampled together but can be at different clocks. Unclassified locked OCT is a monitoring/reconciliation value, not a Vitals health issue and not proof of missing funds.

The global summary row is OCT-denominated and v0 accepts exactly one route. That route must be OCT/wOCT-denominated, and its locked, wrapped, and unclaimed values must match the top-level bridge totals. If future routes include a second route or a non-OCT asset, the row builder must fail until that route has its own family or a denomination-safe aggregate.

## Gateway And UI Contract

`program:update` and `program:submit-snapshot` are dry-run by default. They only sign and submit the state-changing program call when `VITALS_SUBMIT=1`, `VITALS_STATE_PROGRAM_ADDRESS`, `VITALS_OPERATOR_ADDRESS`, and `VITALS_OPERATOR_PRIVATE_KEY_B64` are configured.

When `VITALS_STATE_PROGRAM_ADDRESS` or `VITALS_PROGRAMMED_CIRCLE_ID` is configured, the gateway prefers read-only native reads from the AML state surface and verifies the stored payload/evidence/source-ref hashes before serving `/api/latest`. In the standalone split path this is a `contract_call` read from the Vitals State Program. In the programmed Site Circle path this is a Circle program view read from the Site Circle program.

If `OCTRA_PROGRAM_RPC_URLS` lists more than one endpoint, programmed-Circle latest, history, submit preflight, and submit readback views must agree across all configured program RPCs. `VITALS_MIN_PROGRAM_RPC_URLS` sets the minimum reachable program RPC count and may be `1` when Octra exposes one canonical endpoint. Latest readback comparison includes payload/evidence/source refs, snapshot index, submitter, and the latest summary row/hash; submit preflight compares `get_snapshot_count` before a wallet is loaded. A disagreement fails closed rather than serving a provider-specific view or submitting a new write during RPC lag.

In Circle/static contexts the app should prefer the Circle-local program view adapter and perform the same hash checks in the browser before falling back to the gateway. The older direct `contract_call` adapter remains useful for the devnet split and as a compatibility fallback.

Production/reference mode must set:

```bash
VITALS_STATE_SOURCE_MODE=program_required
```

In `program_required` mode `/api/latest` returns `503` instead of serving gateway-originated observations when the Vitals State Program is unavailable. `/api/history` is sourced only from the AML summary window; if AML history is unavailable, the response is explicitly non-canonical and empty rather than filled from static or local history.

A successful program read reports:

```json
{
  "status": "program",
  "source": "program",
  "authority": {
    "canonical_state_read": true,
    "state_read_from": "vitals-state-program"
  }
}
```

Gateway-originated local bootstrap reads report `status: "bootstrap"` and `source: "bootstrap_live"` so they cannot be mistaken for native state.

`native_readiness.status` is stricter than `status: "program"`. It only reports `native_ready` when the Site Circle is configured, app assets match the pinned Circle release, the active AML target is configured and verified, the AML summary window is readable, and the gateway is running in `program_required` mode. In programmed-Circle mode, the Circle program must expose the required methods, be initialized, have matching source/code/formal-verification artifacts, have expected owner/operator views, meet the configured minimum program-RPC count, and agree across configured program RPCs.

The public Site Circle asset bundle contains app bytes plus `producer.audit.json`, a hash-only audit manifest for the off-chain producer and related source files. It does not contain snapshot/sample/history data. `app/latest_snapshot.sample.json` remains a local/test fixture, but it is not shipped in the Circle manifest and is not a production fallback.
